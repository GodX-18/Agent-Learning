## 开篇：为什么结构决定效果

你有没有遇到过这种情况？

在奶茶店，你对店员说"我要一杯奶茶"，结果上来的可能是：
- 冰的或热的
- 甜的或无糖的
- 加珍珠或不加珍珠的
- 大杯或小杯的

完全随机。你只能将就喝，或者要求重做。

但如果你说"我要一杯大杯、去冰、三分糖、加珍珠的乌龙奶茶"，店员会精确地给你做出来。

**提示词的结构要素，就是你"点奶茶时的规格说明"——说清楚了，AI才能给你"做对"。**

---

## 2.1 指令（Instruction）

### 痛点引入

你请朋友帮忙搬家，说"帮我搬一下东西"，朋友欣然答应。但结果是：他只搬了小箱子，把所有大家具都留下了，因为他说"你说的是东西，我以为你说小物件"。

这个例子说明：**指令不清晰，执行结果就会偏离预期。**

在AI使用中，这种问题更普遍：

```python
# ❌ 模糊指令
prompt = "分析一下这段文字"

# ✅ 清晰指令
prompt = "请分析以下段落的：(1)核心观点 (2)论证结构 (3)语言风格 (4)潜在逻辑漏洞"
```

---

### 生活类比：点菜 vs 开处方

| 对话类型 | 特点 | 结果 |
|----------|------|------|
| "我饿了，给我来点吃的" | 模糊指令 | 完全随机 |
| "我要一份意面，不要辣的" | 明确指令 | 基本可控 |
| "我对海鲜过敏，不能吃虾和蟹，请推荐一道主菜" | 包含约束的指令 | 符合要求 |

**指令的最高境界**：不仅说明"做什么"，还说明"做到什么程度"。

---

### 核心定义

> **指令（Instruction）** 是提示词中明确说明AI需要执行什么任务的核心部分。它告诉AI**目标（Goal）** 和 **动作（Action）** 是什么。

指令的两个维度：
1. **任务类型**：分类、总结、生成、转换、提取、推理、评估...
2. **任务范围**：对什么内容执行这个任务

---

### 落地载体：错误示例与修正对比

```python
# ========== 场景：新闻文章分类 ==========

# ❌ 模糊指令
prompt_v1 = """
请分类这篇文章。
"""

# ✅ 清晰指令
prompt_v2 = """
你是一位新闻分类专家。请将下面的新闻文章分类到以下类别之一：
- 科技 (Technology)
- 财经 (Finance)
- 体育 (Sports)
- 娱乐 (Entertainment)
- 国际 (International)
- 社会 (Society)

输出要求：
- 只输出分类名称，不要其他解释
- 如果无法判断，输出"无法分类"
- 如果属于多个类别，输出主要类别

文章内容：
{news_article}
"""
```

**效果对比**：

| 指标 | 模糊指令 | 清晰指令 |
|------|----------|----------|
| 输出稳定性 | 每次输出格式可能不同 | 固定输出分类名称 |
| 边界处理 | 没有定义"无法判断"怎么办 | 有明确定义 |
| 多标签处理 | 随机选择或全部输出 | 有明确规则 |
| 执行速度 | 快（但返工慢） | 稍慢（但一次做对） |

---

### 边界与陷阱

**陷阱1：指令太长导致"指令稀释"**

有些学员会把大量内容塞进指令部分，导致AI不知道哪个指令更重要。

**生活类比**：老板给你布置任务，一口气说了20件事，你可能只记得最后3件。

**解决方案**：用**清晰的层次结构**区分主指令和辅助信息。用编号、列表、关键词标记。

**陷阱2：指令与上下文混淆**

初学者容易把"背景信息"也放在指令部分，导致AI难以区分"这是什么任务"和"这是什么情况"。

**解决方案**：
- 指令部分只说"做什么"
- 上下文信息单独组织

## 2.2 上下文（Context）

### 痛点引入

你问朋友："今晚吃什么？"

朋友一脸茫然："什么今晚？你请客吗？谁来吃？在哪吃？你有什么忌口吗？"

你愣住了——"就问你今晚吃什么啊！"

朋友说："你突然问我今晚吃什么，我当然要问清楚情况，不然我怎么给你建议？"

**这就是上下文的重要性**：没有上下文，AI只能基于"默认假设"工作，而这些假设很可能与你的真实情况不符。

---

### 生活类比：医生问诊

你去医院看病，医生不会一上来就开药。他会问：

- "哪里不舒服？"（症状描述）
- "持续多久了？"（时间线）
- "之前有类似情况吗？"（历史记录）
- "过敏史有吗？"（特殊情况）

**医生需要上下文来做出准确判断，AI也一样。**

---

### 核心定义

> **上下文（Context）** 是指：为AI提供理解当前任务所需的背景信息、环境条件、约束条件等补充数据。

上下文的价值：
1. **减少臆测**：AI不需要猜测你的情况
2. **个性化输出**：输出与你的具体需求匹配
3. **提高准确性**：基于真实信息而非假设

---

### 落地载体：上下文注入的层次

```python
# ========== 场景：帮用户写商务邮件 ==========

# ❌ 无上下文
prompt_v1 = """
帮我写一封商务邮件。
"""

# ✅ 有上下文
prompt_v2 = """
任务：帮我写一封商务邮件

发件人信息：
- 身份：云服务销售经理
- 公司：XX云
- 经验：5年企业销售

收件人信息：
- 身份：中型科技公司CTO
- 公司：YYY科技
- 关系：初次接触

邮件目的：
- 寻求合作机会
- 预约30分钟电话沟通

邮件要求：
- 语气：专业、简洁、自信
- 长度：200字以内
- 不卑不亢，不要过度热情
"""

# ✅ 过多上下文（误区）
prompt_v3 = """
任务：帮我写一封商务邮件

发件人信息：
- 姓名：张三
- 性别：男
- 年龄：32岁
- 血型：O型
- 星座：射手座
- 爱好：打球、看书
- 养了一只猫叫"小白"
- 猫的年龄：3岁
...（无关信息过多）
"""
```

**上下文质量对比**：

| 版本 | 上下文质量 | 原因 |
|------|-----------|------|
| v1 | 无 | 完全依赖默认假设 |
| v2 | 适量 | 提供任务相关背景 |
| v3 | 过多 | 引入无关噪音，稀释重点 |

---

### 边界与陷阱

**陷阱1：上下文太少——"凭空捏造"**

如果上下文不足，AI会基于训练数据中的常见模式填充空白，可能导致：

- 捏造不存在的公司名、产品名
- 使用不合适的语气（对CEO用对实习生的语气）
- 假设错误的目标受众

**陷阱2：上下文太多——"重点迷失"**

过多上下文会导致：
- AI关注点分散
- Token消耗增加
- 处理速度下降
- 可能产生"上下文污染"（后文信息覆盖前文）

**陷阱3：上下文与指令混淆**

将"背景信息"写在指令位置，或将"任务说明"写在上下文位置。

**正确做法**：
```python
# 推荐结构
prompt = f"""
# 指令（任务说明）
你是一位资深商务邮件撰写专家。请为以下场景撰写邮件。

# 上下文（背景信息）
[发件人背景]
[收件人背景]
[邮件目的]
[特殊约束]

# 输出要求
[格式规范]
"""
```

## 2.3 输入数据（Input）

### 痛点引入

你给朋友发了一条消息："看看这个"——然后附了一张模糊不清的截图。

朋友回复："看什么？"

你才发现截图里内容看不清，而且朋友不知道这是什么东西的截图、为什么要给他看。

**输入数据不清晰，就像给厨师原材料却不说明这是什么原材料、要做成什么菜。**

---

### 生活类比：投喂式与自取式

| 餐厅类型 | 特点 | 对应提示词模式 |
|----------|------|---------------|
| 投喂式 | 你点什么，厨师就做你点的 | 明确输入数据，AI只负责处理 |
| 自取式 | 厨师自己从冰箱里拿食材 | 不给输入数据，AI自己找信息 |

**大多数场景应该用"投喂式"**——你要给AI明确的输入数据。

---

### 核心定义

> **输入数据（Input）** 是指：AI需要处理的原始材料、信息或数据。它回答"处理什么"的问题。

输入数据的类型：
1. **文本内容**：文章、评论、对话、文档等
2. **结构化数据**：JSON、表格、数据库记录等
3. **指令性输入**：用户的具体问题或请求

---

### 落地载体：输入数据的标注与隔离

```python
# ========== 场景：分析商品评论 ==========

# ❌ 输入数据没有隔离
prompt_v1 = """
请分析这条评论的情感：
这个产品真的很好用，物流也很快，就是包装有点简陋。
"""

# ✅ 输入数据清晰标注
prompt_v2 = """
任务：情感分析

输入数据：
```
这个产品真的很好用，物流也很快，就是包装有点简陋。
```

分析要求：
1. 识别评论中的正面词和负面词
2. 判断整体情感（正面/负面/中性）
3. 识别情感强度（强烈/温和）
4. 标注关键转折点
"""

# ✅ 更大规模输入数据的处理
prompt_v3 = """
任务：对以下10条商品评论进行批量情感分析

输入数据（JSON格式）：
```json
[
  {"id": 1, "text": "这个产品真的很好用，物流也很快。", "rating": 5},
  {"id": 2, "text": "一般般，没有宣传的那么好。", "rating": 3},
  {"id": 3, "text": "太差了，用了一周就坏了。", "rating": 1},
  ...（共10条）
]
```

输出要求：
- 对每条评论输出分析结果
- 按以下JSON格式：
```json
[
  {"id": 1, "sentiment": "positive", "keywords": ["很好用", "很快"], "intensity": "strong"},
  ...
]
```
"""
```

---

### 边界与陷阱

**陷阱1：输入数据与上下文混淆**

初学者常把"输入数据"当成"上下文"来处理，或者反过来。

**区分方法**：
- **上下文**：帮助AI理解"为什么"的背景信息
- **输入数据**：AI需要"处理什么"的原材料

**陷阱2：输入数据格式不统一**

当有多条输入数据时，格式不一致会导致AI处理出错。

**解决方案**：使用结构化格式（JSON数组、Markdown表格），并明确说明数据边界。

**陷阱3：输入数据包含隐私信息**

未对输入数据进行脱敏处理，可能导致隐私泄露。

**工程实践**：
```python
def sanitize_input(user_input: str) -> str:
    """
    输入数据脱敏处理
    - 手机号：138****5678
    - 邮箱：u***@example.com
    - 身份证：110***********1234
    """
    import re
    # 手机号脱敏
    user_input = re.sub(r'(\d{3})\d{4}(\d{4})', r'\1****\2', user_input)
    # 邮箱脱敏
    user_input = re.sub(r'(\w+)\@(\w+\.\w+)', r'\1***@\2', user_input)
    return user_input
```

## 2.4 输出格式（Output Format）

### 痛点引入

你去理发店，说"帮我剪个好看的发型"。理发师问"什么格式？"你一脸困惑："剪头发还要什么格式？"

理发师说："我可以剪成短寸、平头、板寸、碎发、偏分...你指哪个？"

你这才意识到——即使同样是"剪头发"，也有无数种"格式"。你的"好看"对他来说毫无意义。

**输出格式，就是让AI的输出从"随机"变成"可控"。**

---

### 生活类比：填写表格 vs 自由写作

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| 填写表格 | 固定字段、结构统一 | 需要程序化处理的场景 |
| 自由写作 | 灵活多样、创意丰富 | 需要创意发挥的场景 |
| 半结构化 | 主体自由+关键字段 | 兼顾灵活性与结构性 |

**工程实践中，90%的场景需要的是半结构化输出**——主体内容灵活，但关键信息位置固定。

---

### 核心定义

> **输出格式（Output Format）** 是指：明确说明AI输出的结构、形式和组织方式。

输出格式的类型：
1. **纯文本**：段落、文章、对话
2. **结构化格式**：JSON、XML、CSV
3. **标记语言**：Markdown表格、HTML
4. **复合格式**：主体文本 + 结构化摘要

---

### 落地载体：输出格式控制的各种模式

```python
# ========== 场景：会议纪要处理 ==========

# ❌ 无格式要求
prompt_v1 = """
请总结以下会议记录的内容。
"""

# ✅ 固定格式输出
prompt_v2 = """
请总结以下会议记录，输出格式如下：

## 会议主题
[一句话描述]

## 参会人员
- [人员1]
- [人员2]

## 关键决议
1. [决议1]
2. [决议2]

## 行动项
| 负责人 | 任务 | 截止日期 |
|--------|------|----------|
| [姓名] | [任务描述] | [日期] |

## 待讨论问题
- [问题1]
- [问题2]

会议记录：
{meeting_notes}
"""

# ✅ JSON格式输出（适合程序处理）
prompt_v3 = """
请提取会议记录的关键信息，输出JSON格式：

{
  "meeting_topic": "string - 会议主题",
  "attendees": ["array - 参会人员列表"],
  "key_decisions": ["array - 关键决议"],
  "action_items": [
    {
      "assignee": "string - 负责人",
      "task": "string - 任务描述",
      "due_date": "string - 截止日期(YYYY-MM-DD格式)"
    }
  ],
  "open_issues": ["array - 待解决问题"]
}

会议记录：
{meeting_notes}

注意：
- 只输出JSON，不要其他内容
- 日期统一为YYYY-MM-DD格式
- 如某字段无法提取，设为null
"""
```

---

### 边界与陷阱

**陷阱1：格式要求与内容质量冲突**

有时候过于严格的格式要求会导致AI为了满足格式而牺牲内容质量。

**解决方案**：
```python
# 优先级声明
prompt = """
输出要求：
1. 【最重要】内容准确、完整
2. 【次重要】遵循以下JSON格式
3. 【约束】如果格式与内容冲突，以内容准确为准，在备注中说明

{
  "summary": "摘要内容（最重要）",
  "details": "详细信息",
  "note": "格式调整说明（如有）"
}
"""
```

**陷阱2：格式过于复杂导致AI理解错误**

尝试输出过于复杂的嵌套结构可能导致格式错误。

**解决方案**：复杂格式分步骤实现，或降低格式复杂度。

**陷阱3：没有定义异常处理**

当输入数据无法满足输出格式要求时，没有定义"怎么办"。

**工程实践**：
```python
def parse_ai_json_output(raw_output: str) -> dict:
    """
    解析AI输出的JSON，包含错误处理
    """
    try:
        import json
        return json.loads(raw_output)
    except json.JSONDecodeError:
        # 尝试提取JSON部分
        import re
        json_match = re.search(r'\{.*\}', raw_output, re.DOTALL)
        if json_match:
            try:
                return json.loads(json_match.group())
            except:
                return {"error": "JSON解析失败", "raw": raw_output}
        return {"error": "未找到JSON", "raw": raw_output}
```

## 2.5 示例（Examples）

### 痛点引入

你教新人写邮件，说"写邮件要专业、简洁、有礼貌"。新人写出来的东西你觉得不专业、不简洁、没礼貌。

你很奇怪——"我都说了要专业简洁有礼貌了啊！"

新人也委屈——"你说的专业是什么标准？简洁是多简洁？有礼貌是要到什么程度？"

这时你可能会说："算了，我给你看几封邮件，你照着这个写。"

**这就是示例的力量——用具体实例代替抽象描述。**

---

### 生活类比：食谱中的"适量"vs 精确配方

| 菜谱类型 | 描述方式 | 学习难度 |
|----------|----------|----------|
| "适量盐，适量油，大火翻炒" | 抽象描述 | 需要大量练习才能掌握 |
| "3克盐，15毫升油，200度翻炒3分钟" | 精确配方 | 按部就班即可完成 |

**示例就像"精确配方"——它告诉AI"按这个样子来"。**

---

### 核心定义

> **示例（Examples）** 是指：在提示词中提供少量具体案例，让AI通过"模仿"来理解任务要求和输出风格。

示例的价值：
1. **降低理解歧义**：具体案例比抽象描述更无歧义
2. **锁定输出模式**：AI会模仿示例的格式、语气、结构
3. **传递隐性知识**：有些"经验"很难用语言描述，但可以通过示例传递

---

### 落地载体：Few-shot提示词设计

```python
# ========== 场景：外卖评论情感分类 ==========

# ❌ Zero-shot（无示例）
prompt_v1 = """
请判断以下外卖评论的情感是正面、负面还是中性。

评论：包装很精美，配送很快，就是味道一般
"""

# ✅ Few-shot（有示例）
prompt_v2 = """
你是一个外卖评论情感分类专家。请判断评论的情感类别。

示例1：
评论："味道超级棒！已经第三次购买了！"
情感：正面
原因：包含"超级棒"、"第三次购买"等强正面信号

示例2：
评论："等了一个半小时才送到，饭都凉了"
情感：负面
原因：等待时间长、食品温度下降是明确负面体验

示例3：
评论："还行吧，没什么特别的"
情感：中性
原因：既无明显赞美也无明显批评

请判断以下评论：
评论："包装很精美，配送很快，就是味道一般"
"""

# ✅ 包含负例的Few-shot
prompt_v3 = """
你是一个外卖评论情感分类专家。请分类以下评论。

正面示例：
- "非常好吃，强烈推荐！" → 正面
- "配送速度快，包装完好" → 正面

负面示例：
- "等了两个小时，饭都凉了" → 负面
- "味道很差，不会再点了" → 负面

中性示例：
- "和上次一样，没什么变化" → 中性
- "中规中矩吧" → 中性

待分类评论：
{review_text}
"""
```

---

### 边界与陷阱

**陷阱1：示例过多导致"示例污染"**

当示例数量过多时，AI可能开始"背诵"示例而非真正理解任务。

**生活类比**：老师给你10道例题让你学习解题方法，和给你1000道例题让你背答案，是完全不同的学习模式。

**经验法则**：通常3-5个示例就能达到良好效果，更多示例边际收益递减。

**陷阱2：示例质量影响输出质量**

如果示例本身有错误或不一致，AI会"学会"这些错误。

**解决方案**：
- 定期审查示例质量
- 确保示例之间格式一致
- 避免示例中出现矛盾

**陷阱3：示例覆盖不足导致"边界情况"失败**

如果只给正面示例，AI可能无法正确处理负面情况。

**解决方案**：确保示例覆盖主要场景类型，包括边界情况。

## 2.6 限制条件（Constraints）

### 痛点引入

你让AI帮你写一篇介绍你们公司产品的文章。结果AI洋洋洒洒写了8000字，从公司创始人童年讲起，一直讲到公司未来十年愿景——就是没提产品。

你很崩溃："我就是让你介绍产品啊！"

AI无辜地说："你只说'介绍产品'，没说介绍多少、介绍哪些方面、多正式..."

**限制条件，就是告诉AI"不要做X"的边界——没有它，AI可能会"过度发挥"。**

---

### 生活类比：装修合同 vs 装修沟通

| 对话形式 | 约束力 | 风险 |
|----------|--------|------|
| "帮我装修一下房子" | 口头，无法律效力 | 完全依赖对方理解 |
| 详细装修合同 | 书面，条款明确 | 有据可依，纠纷可追责 |

**限制条件就是提示词的"合同条款"——明确"不做什么"。**

---

### 核心定义

> **限制条件（Constraints）** 是指：明确说明AI在执行任务时需要遵守的边界、禁区或规则。

限制条件的类型：
1. **内容限制**：不能包含什么、不能提及什么
2. **格式限制**：不能超过什么长度、必须是什么结构
3. **行为限制**：不能做什么操作、必须怎样验证
4. **安全限制**：不能输出什么类型的内容

---

### 落地载体：限制条件的分层设计

```python
# ========== 场景：AI客服回复 ==========

# ❌ 无限制条件
prompt_v1 = """
用户问：你们的退换货政策是什么？
请回答。
"""

# ✅ 有限制条件
prompt_v2 = """
你是一个电商平台的AI客服。请回答用户的退换货政策咨询。

【必须遵守的限制条件】
1. 只能回答退换货相关问题，其他问题回复"这个问题我无法回答，请联系人工客服"
2. 涉及以下关键词必须转人工：投诉、赔偿、律师、法律诉讼
3. 不得承诺任何超出政策范围的优惠
4. 回复总长度不超过100字
5. 不得索要用户密码、验证码等敏感信息

【政策信息】
- 7天无理由退换货（不影响二次销售）
- 质量问题15天内可退换
- 运费险覆盖范围内可申请运费报销

用户问题：你们的退换货政策是什么？
"""

# ✅ 更严格的限制条件（敏感场景）
prompt_v3 = """
你是一个医疗咨询AI。你的角色是"健康科普助手"，不是"医生"。

【严格限制条件】

1. 诊断禁止：
   - 不得给出具体疾病诊断
   - 不得建议具体用药名称和剂量
   - 不得判断疾病严重程度

2. 建议边界：
   - 只提供健康科普信息
   - 所有建议必须包含"建议咨询专业医生"
   - 涉及以下症状必须建议就医：持续高烧、严重出血、意识模糊

3. 紧急情况：
   - 如果用户描述疑似紧急症状，优先回复"请立即拨打120或前往最近急诊"

4. 隐私保护：
   - 不得要求用户提供真实姓名、身份证号
   - 不得存储用户描述的症状信息

用户问题：{user_question}
"""
```

---

### 边界与陷阱

**陷阱1：限制条件太少导致"过度发挥"**

AI在生成内容时会"尽可能扩展"，如果没有限制，它会补充大量可能不需要的信息。

**陷阱2：限制条件太多导致"束手束脚"**

过度限制会限制AI的正常发挥空间，导致输出机械、缺乏有用信息。

**平衡原则**：
- 核心限制必须有（安全、合规、基本质量）
- 次要限制尽量少（保持灵活性）
- 限制条件要明确，不要模糊（如"不要太多"不如"不超过100字"）

**陷阱3：限制条件之间冲突**

同时设置多个限制条件时，可能出现相互矛盾的情况。

**解决方案**：
```python
# 限制条件优先级声明
prompt = """
限制条件优先级（从高到低）：
1. 【最高】安全相关限制（不泄露隐私、不给医疗诊断）
2. 【高】合规相关限制（不违反法律法规）
3. 【中】质量相关限制（内容准确、格式正确）
4. 【低】体验相关限制（语气友好、长度适中）

如果高优先级和低优先级冲突，服从高优先级。
"""
```

├─────────────────────────────────────────────────────────────┤
│                                                             │

│            ┌──────────────┼──────────────┐                  │

│   ┌─────────────────────────────────────────────────────┐  │
│   │                    输出格式 (Output Format)          │  │

│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 六要素的协同原则

1. **指令为核心**：其他要素围绕指令服务
2. **上下文要适量**：足够理解但不过度
3. **输入数据要清晰**：标注隔离，避免混淆
4. **输出格式要明确**：根据用途选择合适格式
5. **示例要精选**：质量比数量重要
6. **限制条件要平衡**：核心限制必须有，避免过度约束

### 第一题：概念理解
> 提示词的六大结构要素是什么？请用自己的语言解释每个要素的核心作用。

<details>
<summary><b>参考答案（点击展开）</b></summary>

**参考答案**：

1. **指令（Instruction）**：告诉AI"做什么"和"做到什么程度"——核心任务描述
2. **上下文（Context）**：提供理解任务所需的背景信息——回答"为什么"
3. **输入数据（Input）**：AI需要处理的原始材料——回答"处理什么"
4. **输出格式（Output Format）**：AI输出的结构规范——回答"怎么输出"
5. **示例（Examples）**：通过案例演示期望模式——回答"照着做"
6. **限制条件（Constraints）**：明确不能做什么的边界——回答"不能做X"

六大要素协同的核心逻辑：指令是核心，上下文提供背景，输入数据提供原料，示例提供模式参考，输出格式规范结果，限制条件划定边界。

# 2. 没有输出格式：可能每次输出格式都不一样
# 3. 没有限制条件：可能输出过长或过短
# 4. 上下文不足：没有说明是什么类型的新闻
```

<details>
<summary><b>参考答案（点击展开）</b></summary>

**参考答案**：

**问题清单**：

| 问题 | 严重程度 | 原因 |
|------|----------|------|
| 指令模糊 | 高 | "简洁""全面""有深度"无法量化，AI理解可能差异很大 |
| 无输出格式 | 中 | 无法程序化处理，每次输出结构可能不同 |
| 无限制条件 | 中 | 可能输出过长（500字+）或过短（50字） |
| 无上下文 | 低 | 新闻内容已提供，但可以补充新闻类型 |

**修改方案**：

```python
prompt = """
你是一个新闻摘要专家。请为以下新闻生成简短摘要。

【新闻背景】
- 新闻类型：{news_type}
- 目标读者：普通公众

【输入数据】
{news_content}

【指令】
生成一段100-150字的中文摘要

【输出格式】
必须包含以下三部分：
1. 一句话核心信息（不超过30字）
2. 关键细节（3-5个要点，用•标记）
3. 影响/意义分析（1-2句话）

【限制条件】
- 只基于新闻内容总结，不要添加外部信息
- 使用客观中立的语言
- 避免主观评价词汇（"令人震惊"、"难以置信"等）
"""

# 修改要点：
# 1. 明确了"100-150字"的可量化标准
# 2. 添加了结构化的输出格式（三部分）
# 3. 添加了具体的限制条件（不添加外部信息、客观语言）
# 4. 补充了上下文（新闻类型、目标读者）
```
